## 1. Навигация и информационная архитектура

### 1.1 Sidebar — **Важно**
**Проблема:** Всего 2 пункта навигации (Дашборд, Проекты). Sidebar занимает 240px, но несёт мало информации.
**Рекомендация:**
- Добавить "Недавние проекты" (3-5 последних) для быстрого доступа
- Добавить счётчик непрочитанных уведомлений на колокольчик
- Рассмотреть коллапс до иконок (60px) на десктопе для экономии места

### 1.2 Вложенная навигация — **Улучшение**
**Проблема:** При переходе в проект → визит — нет подсветки текущего местоположения в sidebar. Пользователь теряет контекст.
**Рекомендация:**
- Подсвечивать "Проекты" при просмотре любого проекта/визита
- Показывать мини-хлебные крошки в sidebar (Project > Visit #)

### 1.3 Вкладки проекта — **ОК**
6 вкладок с иконками и текстом — хорошая структура. RBAC-фильтрация прозрачна (пользователь не видит того, что ему недоступно). Горизонтальный скролл на мобиле работает.

### 1.4 Возврат назад — **Улучшение**
**Проблема:** Хлебные крошки работают, но при возврате из визита в проект теряется позиция скролла и активная вкладка (всегда "Обзор").
**Рекомендация:** Запоминать последнюю вкладку проекта при возврате.

## 2. Формы и ввод данных

### 2.1 Валидация форм — **ОК**
Инлайн-валидация реализована: красные рамки, текст ошибки под полем. Работает на:
- Создание проекта (название обязательно)
- Создание визита (название + дата)
- Выставление счёта (название + сумма > 0)
- Приглашение участника (email-формат)

### 2.2 Дата-пикер — **Важно**
**Проблема:** HTML `<input type="date">` — нативный пикер без ограничений. Можно:
- Запланировать визит в прошлом
- Выставить счёт с прошедшим сроком оплаты
**Рекомендация:** Установить `min` на сегодняшнюю дату для планируемых визитов и сроков оплаты.

### 2.3 Загрузка файлов — **ОК**
Drag-and-drop + клик работают. Предпросмотр фото отображается. Ограничения:
- Фото: 20 МБ, jpg/png/webp
- Документы: 50 МБ, pdf/dwg/xlsx/png

**Проблема (Важно):** Нет прогресс-бара при загрузке. На медленном соединении пользователь не знает, загружается ли файл.
**Рекомендация:** Добавить progress bar (Supabase Storage поддерживает onUploadProgress).

### 2.4 Модалки — **Улучшение**
**Проблема:** При закрытии модалки (клик по фону или кнопка) — данные теряются без предупреждения. Если пользователь заполнил форму и случайно закрыл — начинает сначала.
**Рекомендация:** Если форма заполнена — запросить подтверждение: "Отменить заполнение?"

### 2.5 Ссылка на оплату — **ОК**
Новое поле с `type="url"` + плейсхолдер. В списке счетов отображается как кликабельная ссылка "Оплатить". Хорошая UX-реализация.

## 3. Обратная связь и состояния

### 3.1 Toast-уведомления — **Важно**
**Проблема:** Показывается только 1 toast. Если пользователь быстро делает 2 действия — видит только последнее.
**Рекомендация:** Очередь toast (стек из 3-5), каждый с авто-закрытием через 4 сек.

### 3.2 Загрузка данных — **Важно**
**Проблема:** Единый спиннер "Загрузка..." для всех состояний. Нет скелетон-лоадеров. При переключении вкладок — кратковременный мерцающий спиннер.
**Рекомендация:**
- Skeleton screens для карточек и списков
- Кэширование загруженных данных (React Query / SWR)
- Fade-transition между вкладками

### 3.3 Пустые состояния — **ОК**
Реализованы для:
- Нет проектов → "Создайте первый проект"
- Нет визитов → текст-заглушка
- Нет счетов → "Счетов пока нет"
- Нет фото → текст

**Улучшение:** Добавить иллюстрации (не просто текст) и конкретные CTA-кнопки.

### 3.4 Ошибки — **Критично**
**Проблема:** Ошибки Supabase показываются как есть — на английском, технический текст:
- "new row violates row-level security policy"
- "JSON object requested, multiple (or no) rows returned"

**Рекомендация:** Маппинг кодов ошибок Supabase в человекопонятные сообщения на русском.

### 3.5 Состояние кнопок — **ОК**
Кнопки "Сохранить" / "Загрузить" → disabled + "Сохранение..." при отправке. Хорошо.

## 4. Визуальная иерархия и типографика

### 4.1 Типографика — **ОК**
- Шрифт DM Sans — современный, читаемый
- JetBrains Mono для чисел — хороший контраст
- Размеры: заголовки 17px, текст 13px, мелкий 11px

### 4.2 Цветовая система — **Важно**
**Проблема:** Цвета хардкожены в компонентах (`text-[#E85D3A]`, `bg-[#FEF0EC]`), а не в CSS-переменных. Это:
- Затрудняет поддержку единообразия
- Блокирует тёмную тему
- Усложняет ребрендинг

**Рекомендация:** Вынести в Tailwind config или CSS custom properties:
```css
--color-danger: #E85D3A;
--color-success: #2A9D5C;
--color-warning: #D4930D;
```

### 4.3 Статус-бейджи — **ОК**
Цветные бейджи для всех типов статусов (фото, supply, риски). Единообразный паттерн через `PHOTO_STATUS_CONFIG`, `SUPPLY_STATUS_CONFIG`, `RISK_CONFIG`.

### 4.4 Иконки — **ОК**
28 SVG-иконок, все в одном файле. Единый стиль: stroke, 1.8px, round caps. Хороший набор для MVP.

### 4.5 Карточки — **ОК**
Единый `.card` стиль: белый фон, тень, скруглённые углы, hover-эффект (border color). Хорошая визуальная группировка.

## 5. Мобильная адаптация

### 5.1 Sidebar — **ОК**
На мобиле: overlay с анимацией slide-in, затемнение фона, кнопка-гамбургер. Работает.

### 5.2 Вкладки — **ОК**
Горизонтальный скролл с скрытым скроллбаром. Работает, но:
**Улучшение:** Добавить fade-эффект на краях (gradient mask), чтобы показать что есть ещё вкладки.

### 5.3 Модалки — **ОК**
На мобиле модалки отображаются как bottom-sheet (прижаты к низу экрана). Хороший паттерн.

### 5.4 Tap targets — **ОК**
Минимальный размер 44px для кнопок и ссылок. Соответствует WCAG.

### 5.5 Таблицы supply — **Важно**
**Проблема:** Таблица спецификации (SupplySpec) не адаптирована для мобиле. Горизонтальный скролл работает, но:
- Колонки слишком узкие
- Текст обрезается
- Dropdown-статус маленький для пальца

**Рекомендация:** На мобиле заменить таблицу на список карточек (card view).

### 5.6 Gantt-таймлайн — **Важно**
**Проблема:** Timeline горизонтальный — на мобиле неюзабелен (нужно скроллить в обе стороны).
**Рекомендация:** На мобиле показывать вертикальный список вместо Gantt.

## 6. Доступность (Accessibility)

### 6.1 ARIA-атрибуты — **Критично**
**Проблема:** Модалки не имеют `role="dialog"`, `aria-modal`, `aria-labelledby`. Табы не имеют `role="tablist"`, `role="tab"`, `aria-selected`. Screen reader не сможет интерпретировать интерфейс.

### 6.2 Фокус-менеджмент — **Критично**
**Проблема:** При открытии модалки фокус не переносится внутрь. При закрытии — не возвращается к кнопке-триггеру. Tab-навигация уходит за модалку (нет focus trap).

### 6.3 Цветовая индикация — **Важно**
**Проблема:** Статусы фото (замечание/принято) различаются только цветом (красный/зелёный). Для дальтоников (8% мужчин) — неразличимы.
**Рекомендация:** Добавить иконки к статусам: замечание (⚠), принято (✓), в работе (⟳).

### 6.4 Alt-тексты — **Важно**
**Проблема:** Загруженные фото не имеют alt-тексов. Иконки-кнопки не имеют aria-label.
**Рекомендация:**
```jsx
<img alt={photo.comment || `Фото: ${photo.zone || 'без зоны'}`} />
<button aria-label="Редактировать название проекта">
```

### 6.5 Клавиатура — **Улучшение**
- Escape закрывает модалки — **ОК**
- Enter подтверждает — **ОК**
- Tab-навигация между элементами — **Не проверено, вероятно проблемы**

## 7. Целевые сценарии (User Flows)

### 7.1 Сценарий: Дизайнер создаёт проект с нуля — **ОК**
```
Вход → Дашборд → "+ Новый проект" → Заполнение → Сохранить → Проект создан
```
Работает. 3 клика до создания проекта. Хороший onboarding flow.

### 7.2 Сценарий: Визит на объект — **ОК**
```
Проект → Визиты → "Запланировать" → Форма → Создано
→ Открыть визит → "Добавить фото" → Drag файл → Заполнить → Загрузить
```
Работает. 5-6 кликов для загрузки фото. Приемлемо.

### 7.3 Сценарий: Выставить счёт — **ОК**
```
Проект → Journal → "Выставить счёт" → Форма (+ ссылка) → Сохранить
```
Работает. Ссылка на оплату — удобно для клиента.

### 7.4 Сценарий: Приглашение клиента — **Важно**
```
Проект → Настройки → "Пригласить" → По ссылке → Скопировать → Отправить клиенту
Клиент: переход по ссылке → Регистрация → Автодобавление в проект
```
**Проблема:** При переходе по ссылке клиент видит стандартную страницу входа без объяснения, что его пригласили. Нет сообщения "Вас пригласили в проект X".
**Рекомендация:** Показать информацию о проекте и пригласившем перед/после входа.

### 7.5 Сценарий: Импорт Excel — **ОК**
```
Проект → Supply → Импорт → Загрузить файл → Маппинг → Превью → Импорт
```
4-шаговый визард — хороший паттерн. Авто-маппинг колонок экономит время.

### 7.6 Сценарий: Клиент проверяет статус — **Улучшение**
```
Вход (как client) → Проект → Обзор (KPI) → Визиты (смотрит фото) → Journal (смотрит счета)
```
Работает, но:
- Клиент не видит модуль Supply (правильно по RBAC)
- Клиент не может оплатить счёт напрямую (только по ссылке)
- Нет чата/комментариев для коммуникации с дизайнером

## Общая оценка

**MVP-уровень: 7/10**

Сильные стороны:
- Чистый, минималистичный дизайн
- Хорошая RBAC-система (скрытие вместо запрета)
- Функциональный supply-модуль с Gantt и импортом
- Адекватная мобильная адаптация (sidebar, bottom-sheet)
- Drag-and-drop загрузка файлов

Слабые стороны:
- Отсутствие accessibility (a11y) — барьер для слабовидящих пользователей
- Технические ошибки Supabase показываются "как есть"
- Нет прогресса загрузки файлов
- Supply-таблица неюзабельна на мобиле
- Нет чата/комментариев (ключевой pain point для коммуникации дизайнер-клиент)
